Window Functions — Senior Level

Core Anatomy

Every window function has three optional clauses. Mastering them is table-stakes at senior level.

FUNC(col) OVER (
  PARTITION BY group_col      -- reset per group
  ORDER BY     sort_col       -- defines frame order
  ROWS BETWEEN UNBOUNDED PRECEDING
           AND CURRENT ROW   -- frame clause
)
Without ORDER BY, the default frame is the ENTIRE PARTITION — not just preceding rows. This surprises most candidates.
The Ranking Trinity
-- RANK: gaps on ties (1,1,3)
RANK() OVER (ORDER BY score DESC)

-- DENSE_RANK: no gaps (1,1,2)
DENSE_RANK() OVER (ORDER BY score DESC)

-- ROW_NUMBER: always unique (1,2,3)
ROW_NUMBER() OVER (ORDER BY score DESC)

-- ✅ De-dup pattern (keep 1 per group)
SELECT * FROM (
  SELECT *,
    ROW_NUMBER() OVER
      (PARTITION BY user_id ORDER BY created_at DESC) rn
  FROM orders
) WHERE rn = 1
LAG / LEAD — Time Series
-- Day-over-day revenue delta
SELECT
  date,
  revenue,
  LAG(revenue, 1, 0) OVER (ORDER BY date) prev_rev,
  revenue - LAG(revenue) OVER (ORDER BY date) AS delta
FROM daily_sales;

-- 3rd arg = default when no prev row exists
⚠ LAG/LEAD default 3rd arg is NULL if omitted — always set it explicitly to avoid NULL math errors.
Frame Clause Mastery
-- 7-day rolling average
AVG(revenue) OVER (
  ORDER BY date
  ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
)

-- RANGE vs ROWS — critical difference:
-- RANGE: uses logical value range (peers share frame)
-- ROWS : uses physical row offset (always exact)

-- Cumulative sum
SUM(amount) OVER (
  PARTITION BY user_id
  ORDER BY created_at
  ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
) AS running_total
⚠ WINDOW TRAP: WHERE Cannot Filter WindowsTRAP

Window functions are computed AFTER WHERE but BEFORE the outer SELECT. You cannot filter on a window result in the same query level.

-- ❌ WRONG — will throw error
SELECT *, ROW_NUMBER() OVER (ORDER BY id) rn
FROM t
WHERE rn = 1;  -- column "rn" does not exist

-- ✅ CORRECT — wrap in subquery or CTE
WITH ranked AS (
  SELECT *, ROW_NUMBER() OVER (ORDER BY id) rn
  FROM t
)
SELECT * FROM ranked WHERE rn = 1;
💡 Gaps & Islands — Classic Interview ProblemTRICK

Find consecutive sequences (sessions, streaks, contiguous dates).

-- Island = group of consecutive dates
SELECT
  MIN(dt) island_start,
  MAX(dt) island_end,
  COUNT(*) days
FROM (
  SELECT
    dt,
    -- subtract row number from date to get "island id"
    dt - ROW_NUMBER() OVER (ORDER BY dt)
       ::int AS grp
  FROM active_days
) t
GROUP BY grp
ORDER BY island_start;

⚠ In PostgreSQL, CTEs used to be optimization fences (always materialized). Since PG 12, the planner can inline them. Use MATERIALIZED or NOT MATERIALIZED hints explicitly when it matters.
Recursive CTE — Org Hierarchy
WITH RECURSIVE org AS (
  -- 1. Anchor: root node(s)
  SELECT id, name, manager_id, 0 AS depth
  FROM employees
  WHERE manager_id IS NULL

  UNION ALL

  -- 2. Recursive: join to prior result
  SELECT e.id, e.name, e.manager_id, o.depth+1
  FROM employees e
  JOIN org o ON e.manager_id = o.id
)
SELECT * FROM org ORDER BY depth, name;
⚠ Always add a depth or cycle guard — infinite loops are possible with circular foreign keys. PostgreSQL 14+ supports CYCLE clause natively.
💡 CTE as a Step-DebuggerTRICK

Break complex analytics into named, testable steps. Each CTE is independently verifiable — rename it to the final SELECT to debug intermediate state.

WITH
  raw        AS (SELECT ... FROM events),
  filtered   AS (SELECT ... FROM raw  WHERE ...),
  aggregated AS (SELECT ... FROM filtered GROUP BY ...),
  ranked     AS (SELECT *, ROW_NUMBER() OVER (...) rn
                 FROM aggregated)
SELECT * FROM ranked WHERE rn = 1;

-- Debug: temporarily swap final SELECT to:
-- SELECT * FROM filtered;  ← see intermediate

⚠ TRAP: Row Multiplication on 1-to-Many JOINTRAP
-- orders (1) → order_items (many)
-- This MULTIPLIES rows — SUM is wrong!
SELECT
  o.order_id,
  SUM(o.discount)  -- ❌ discount counted N times
FROM orders o
JOIN order_items oi ON o.order_id = oi.order_id
GROUP BY o.order_id;

-- ✅ Aggregate BEFORE joining
WITH item_totals AS (
  SELECT order_id, SUM(price) total
  FROM order_items GROUP BY order_id
)
SELECT o.order_id, o.discount, it.total
FROM orders o
JOIN item_totals it ON o.order_id = it.order_id;
⚠ TRAP: Filtering LEFT JOIN in WHERETRAP
-- Intent: all users + their orders if any
-- ❌ WHERE turns LEFT JOIN into INNER JOIN
SELECT u.*, o.order_id
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE o.status = 'paid';  -- kills NULLs = inner join

-- ✅ Move filter into ON clause
SELECT u.*, o.order_id
FROM users u
LEFT JOIN orders o
  ON u.id = o.user_id AND o.status = 'paid';
LATERAL JOIN / CROSS APPLY

LATERAL lets the right side of a join reference columns from the left side — it's a correlated subquery as a join.

-- Top 3 orders per user (efficient!)
SELECT u.id, u.name, latest.*
FROM users u
JOIN LATERAL (
  SELECT *
  FROM orders
  WHERE user_id = u.id  -- ← references outer table
  ORDER BY created_at DESC
  LIMIT 3
) latest ON TRUE;
Use CROSS APPLY in SQL Server / Oracle. MySQL 8.0.14+ supports LATERAL but with restrictions.

⚠ TRAP: Index Killer PatternsTRAP
-- ❌ Function on indexed column = full scan
WHERE LOWER(email) = 'foo@bar.com'
-- ✅ Use functional index or citext
CREATE INDEX ON users (LOWER(email));

-- ❌ Leading wildcard = full scan
WHERE name LIKE '%smith'
-- ✅ Trailing wildcard uses index
WHERE name LIKE 'smith%'

-- ❌ Implicit cast = index not used
WHERE user_id = '123'  -- user_id is INT
-- ✅ Match types
WHERE user_id = 123
💡 Reading EXPLAIN ANALYZETRICK
EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON)
SELECT * FROM orders WHERE user_id = 42;
  • rows= vs actual rows=: large discrepancy → stale stats, run ANALYZE
  • Seq Scan on large table: missing index or planner chose it (check row count estimate)
  • Buffers: hit=X read=Y: Y reads from disk → consider caching or I/O tuning
  • Sort (external merge): hit work_mem limit → increase session work_mem
  • Read plans bottom-up: innermost node executes first
Composite Index Column Order

The leftmost prefix rule: a composite index (a, b, c) supports queries on a, (a,b), (a,b,c) — but NOT b or c alone.

CREATE INDEX idx_user_date
  ON orders (user_id, created_at);

-- ✅ Uses index (leading column)
WHERE user_id = 1
WHERE user_id = 1 AND created_at > '2024-01-01'

-- ❌ Does NOT use index (skips leading col)
WHERE created_at > '2024-01-01'
Rule of thumb: highest-cardinality equality column first, then range columns.

NULL — The Silent Killer

⚠ NULL in AggregatesTRAP
-- COUNT(*) vs COUNT(col)
COUNT(*)      -- counts all rows
COUNT(col)  -- skips NULLs ← silent!

-- AVG also silently skips NULLs
-- AVG(col) ≠ SUM(col)/COUNT(*)
-- when col has NULLs

-- ✅ Explicit handling
AVG(COALESCE(col, 0))
⚠ NOT IN with NULLsTRAP
-- If subquery returns ANY NULL,
-- NOT IN returns 0 rows — always!
WHERE id NOT IN
  (SELECT user_id FROM orders)
-- ❌ Returns nothing if any user_id is NULL

-- ✅ Use NOT EXISTS instead
WHERE NOT EXISTS (
  SELECT 1 FROM orders
  WHERE orders.user_id = users.id
)
⚠ NULL Comparison is Always UnknownTRAP
-- NULL = NULL → UNKNOWN (not TRUE!)
-- NULL != NULL → UNKNOWN

-- ❌ Always 0 rows
WHERE col = NULL
WHERE col != NULL

-- ✅ Correct
WHERE col IS NULL
WHERE col IS NOT NULL

-- ✅ NULL-safe equality (PG 15+)
WHERE col IS NOT DISTINCT FROM other

Aggregation Traps & Advanced Grouping

⚠ HAVING vs WHERE Execution OrderTRAP

SQL logical order: FROM → JOIN → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT

-- WHERE: filters BEFORE grouping
-- HAVING: filters AFTER grouping (on aggregates)

-- ❌ Alias in HAVING (most DBs reject)
SELECT dept, SUM(salary) total
FROM emp
GROUP BY dept
HAVING total > 100000;  -- alias not visible

-- ✅ Repeat the expression
HAVING SUM(salary) > 100000
Performance: always push non-aggregate filters into WHERE, not HAVING. WHERE reduces rows before grouping.
GROUPING SETS / ROLLUP / CUBE
-- ROLLUP: subtotals + grand total
SELECT year, quarter, SUM(revenue)
FROM sales
GROUP BY ROLLUP(year, quarter);
-- Generates: (year,qtr), (year), () groups

-- CUBE: all combinations
GROUP BY CUBE(region, product, year)
-- 2^3 = 8 grouping combinations

-- GROUPING SETS: explicit combinations
GROUP BY GROUPING SETS (
  (region, product),
  (region),
  ()
)
Use GROUPING(col) function to distinguish NULL-from-rollup vs actual NULL data in result rows.
💡 Conditional Aggregation — Pivot Without PIVOTTRICK
-- Monthly revenue by category in one row per month
SELECT
  DATE_TRUNC('month', created_at) month,
  SUM(CASE WHEN category='A' THEN amount ELSE 0 END) cat_a,
  SUM(CASE WHEN category='B' THEN amount ELSE 0 END) cat_b,
  -- Modern syntax (PostgreSQL 9.4+):
  SUM(amount) FILTER (WHERE category='C') cat_c
FROM orders
GROUP BY 1;

Transactions, Locking & Isolation

Isolation Levels vs Anomalies
LevelDirty ReadNon-RepeatablePhantom
READ UNCOMMITTED✓ Yes✓ Yes✓ Yes
READ COMMITTED✗ No✓ Yes✓ Yes
REPEATABLE READ✗ No✗ No✓ Yes
SERIALIZABLE✗ No✗ No✗ No
PostgreSQL's default is READ COMMITTED. Its REPEATABLE READ also prevents phantoms (MVCC-based). MySQL's REPEATABLE READ prevents phantoms too with gap locks.
⚠ TRAP: Deadlock PatternTRAP
-- Session A                 Session B
BEGIN;                      BEGIN;
UPDATE accounts            UPDATE accounts
  SET bal=bal-100           SET bal=bal+200
  WHERE id=1;  ← locks 1     WHERE id=2; ← locks 2

UPDATE accounts            UPDATE accounts
  WHERE id=2;  ← waits       WHERE id=1; ← waits
-- DEADLOCK DETECTED → one tx rolled back
✅ Fix: always acquire locks in a consistent global order (always lock id=1 before id=2). Or use SELECT FOR UPDATE SKIP LOCKED for queue patterns.
💡 SELECT FOR UPDATE SKIP LOCKED — Job QueueTRICK
-- Safe concurrent job queue — no double-processing
BEGIN;

SELECT id, payload
FROM job_queue
WHERE status = 'pending'
ORDER BY created_at
LIMIT 1
FOR UPDATE SKIP LOCKED;  -- skip rows locked by peers

UPDATE job_queue
  SET status = 'processing'
WHERE id = :claimed_id;

COMMIT;
⚠ SAVEPOINT for Partial Rollback
BEGIN;
  INSERT INTO orders ...;

  SAVEPOINT before_items;

  INSERT INTO order_items ...;
  -- something goes wrong
  ROLLBACK TO SAVEPOINT before_items;

  -- retry with corrected data
  INSERT INTO order_items ...;
COMMIT;
-- Order still committed, items retried

Performance Tricks

💡 EXISTS vs COUNT — Early ExitTRICK
-- ❌ Scans entire table
IF (SELECT COUNT(*) FROM orders
    WHERE user_id=42) > 0

-- ✅ Stops at first match
IF EXISTS (
  SELECT 1 FROM orders
  WHERE user_id=42
)
💡 Batch DELETE / UPDATE — Avoid Lock StormsTRICK
-- ❌ Deletes 10M rows in one lock-holding tx
DELETE FROM events WHERE created_at < '2020-01-01';

-- ✅ Loop in batches (application layer or DO loop)
DO $$
DECLARE deleted INT;
BEGIN
  LOOP
    DELETE FROM events
    WHERE id IN (
      SELECT id FROM events
      WHERE created_at < '2020-01-01'
      LIMIT 1000
    );
    GET DIAGNOSTICS deleted = ROW_COUNT;
    EXIT WHEN deleted = 0;
    PERFORM pg_sleep(0.01);  -- breathe
  END LOOP;
END$$;
💡 Covering Index — Zero Heap AccessTRICK
-- Query needs: user_id (filter), email (return)
-- Normal index on user_id → index scan + heap fetch
-- Covering index → index-only scan (no heap)

CREATE INDEX idx_covering
ON users (user_id)
INCLUDE (email, name);  -- PG 11+

-- EXPLAIN shows: Index Only Scan ← ideal
✅ Include columns that appear in SELECT but not WHERE/ORDER BY. Don't add too many — increases index size and write cost.
UPSERT Patterns
-- PostgreSQL
INSERT INTO user_stats (user_id, logins)
VALUES (42, 1)
ON CONFLICT (user_id)
DO UPDATE SET
  logins = user_stats.logins + EXCLUDED.logins,
  updated_at = NOW();

-- MySQL
INSERT INTO user_stats ...
ON DUPLICATE KEY UPDATE
  logins = logins + VALUES(logins);

-- EXCLUDED refers to the proposed insert row

JSON, Arrays & Modern SQL

PostgreSQL JSONB Essentials
-- → returns JSONB, ->> returns TEXT
data->'user'->>'email'

-- ? operator: key exists
WHERE metadata ? 'premium'

-- @> containment
WHERE tags @> '["vip","active"]'::jsonb

-- jsonb_array_elements: unnest JSON array
SELECT id, jsonb_array_elements(items) item
FROM orders;

-- Update nested key immutably
UPDATE users
SET profile =
  profile || '{"verified": true}'::jsonb
WHERE id = 42;
Use GIN index on JSONB for ?, @>, @@ operators. Use expression index for frequent ->> lookups.
UNNEST, ANY, ALL — Array Power
-- ANY: value matches any in array
WHERE status = ANY(ARRAY['active','trial'])
-- equivalent to IN but typed

-- ALL: must match all
WHERE price < ALL(ARRAY[10, 20, 30])

-- UNNEST: expand array to rows
SELECT id, UNNEST(tags) AS tag
FROM articles;

-- Array aggregation
SELECT user_id,
  ARRAY_AGG(product_id ORDER BY created_at) products
FROM orders
GROUP BY user_id;

Last-Minute Interview Prep — 20 Senior Questions

What is the difference between RANK(), DENSE_RANK(), and ROW_NUMBER()?
All three rank rows within a partition. ROW_NUMBER always returns unique sequential integers regardless of ties. RANK gives the same rank to ties but skips the next rank (1,1,3). DENSE_RANK gives the same rank to ties with no gaps (1,1,2). Use ROW_NUMBER for de-duplication, DENSE_RANK when downstream code needs contiguous rank values.
What happens when you use NOT IN and the subquery contains a NULL?
The result is always empty (0 rows). NULL comparisons return UNKNOWN, and x NOT IN (..., NULL, ...) evaluates to NOT (x=1 OR x=NULL OR ...) = NOT (FALSE OR UNKNOWN) = NOT UNKNOWN = UNKNOWN. Always prefer NOT EXISTS which is NULL-safe and often faster via index.
What is the SQL logical order of operations?
FROM → JOIN → WHERE → GROUP BY → HAVING → SELECT → DISTINCT → ORDER BY → LIMIT/OFFSET. This explains why you can't use a SELECT alias in WHERE or HAVING — the alias doesn't exist yet at that stage. It also explains why window functions can't be filtered in the same SELECT level.
What is the difference between a clustered and non-clustered index?
A clustered index defines the physical storage order of the table's data (only one allowed per table — the primary key in most systems). A non-clustered index is a separate B-tree structure with pointers back to the heap/data pages. In PostgreSQL, all indexes are non-clustered by default; use CLUSTER command to physically reorder once, but it doesn't stay clustered on inserts.
Explain MVCC and why it matters.
Multi-Version Concurrency Control means each transaction sees a snapshot of the database at transaction start. Writers create new row versions (with xmin/xmax visibility metadata) rather than overwriting in-place. Readers never block writers and writers never block readers. This enables high concurrency but requires a vacuum process to reclaim dead row versions. PostgreSQL and MySQL InnoDB both use MVCC.
When would you use a partial index?
When queries consistently filter on a subset of rows. Example: CREATE INDEX ON orders (user_id) WHERE status = 'pending' — if 95% of queries only look at pending orders, a partial index is much smaller (faster scans, less memory) and only needs to be maintained for writes that match the predicate. Classic use case: soft-delete tables where you always add WHERE deleted_at IS NULL.
What are phantom reads and which isolation level prevents them?
A phantom read occurs when a transaction re-executes a range query and sees new rows inserted by another committed transaction. SERIALIZABLE isolation prevents phantoms. REPEATABLE READ in the SQL standard does not prevent them, but PostgreSQL's MVCC-based implementation also prevents phantoms at REPEATABLE READ level in practice.
How would you find duplicate records and delete all but one?
Use ROW_NUMBER() partitioned by the duplicate key, then delete rows where rn > 1: DELETE FROM t WHERE id IN (SELECT id FROM (SELECT id, ROW_NUMBER() OVER (PARTITION BY email ORDER BY id) rn FROM t) x WHERE rn > 1). Always preview with a SELECT first. In high-traffic tables, do it in batches.
Explain the difference between correlated and non-correlated subqueries, and their performance implications.
A non-correlated subquery executes once and its result is reused (acts like a constant). A correlated subquery references the outer query and re-executes for every outer row — O(n) executions. Correlated subqueries can be very slow on large tables. Most can be rewritten as a JOIN or window function. EXISTS with correlated subquery is an exception — it's usually efficient due to short-circuit evaluation.
What is a covering index and when should you use it?
A covering index includes all columns needed by a query — the filter columns AND the returned columns — so the engine never needs to touch the heap/data pages (Index Only Scan). Use when a query runs very frequently and the extra index size (storage + write overhead) is justified. In PostgreSQL, use the INCLUDE clause to add non-key columns without affecting sort order.
How do you approach optimizing a slow query in production?
1) Run EXPLAIN (ANALYZE, BUFFERS) — check for Seq Scans on large tables, row count estimate vs actual discrepancy, and slow sort/hash nodes. 2) Check index usage — is the right index present and being used? 3) Check statistics freshness (ANALYZE). 4) Rewrite joins, eliminate correlated subqueries, push filters earlier. 5) Consider materialized views or partial indexes for repeated heavy aggregations.
What's the difference between UNION and UNION ALL?
UNION ALL returns all rows from both queries including duplicates — O(n). UNION deduplicates the combined result set — O(n log n) sort/hash. Always use UNION ALL when you know there are no duplicates or duplicates are acceptable; it's significantly faster. The deduplication in UNION has the same performance characteristics as a DISTINCT operation.
What is a materialized view and when would you use it?
A materialized view stores the query result set physically on disk, unlike a regular view which re-executes the query each time. Use when: the underlying query is expensive (multi-table aggregations, complex joins), the data doesn't need to be real-time, and reads far outnumber the acceptable refresh cost. Refresh with REFRESH MATERIALIZED VIEW CONCURRENTLY in PostgreSQL to avoid locking reads during refresh.
Explain the difference between optimistic and pessimistic locking.
Pessimistic locking locks the row on read (SELECT FOR UPDATE), preventing other writers until committed — safe but reduces concurrency. Optimistic locking doesn't lock on read; instead it adds a version column checked at update time (WHERE version = :seen_version). If the check fails, the application retries. Better for low-conflict scenarios; pessimistic is better when conflicts are frequent or the window between read and write is long.
How would you implement a running total / cumulative sum?
Use SUM(amount) OVER (PARTITION BY user_id ORDER BY created_at ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW). The explicit ROWS BETWEEN frame clause is important — without it, same-value ORDER BY rows may form a peer group where all peers are summed together, giving incorrect results for ties.
What is table partitioning and what problems does it solve?
Partitioning divides a large table into smaller physical segments (partitions) based on a column value (range, list, or hash). Benefits: partition pruning (planner only scans relevant partitions), easier archival/deletion (drop a partition vs batch delete), and potential parallel scan improvements. Downsides: cross-partition queries can be slower, indexes must exist per partition, and foreign keys have limitations.
What's the danger of SELECT * in production queries?
1) Schema changes (added/renamed columns) can silently break application code. 2) Prevents covering index scans — forces heap access for all columns. 3) Transfers unnecessary data over the network. 4) In JOINs, ambiguous column names can cause silent errors. Always project only the columns you need.
How does a hash join work and when does it beat a nested loop?
A hash join builds an in-memory hash table from the smaller relation (build phase), then probes it with each row from the larger relation (probe phase). Time complexity O(n+m). It beats nested loops when: (a) there's no useful index on the inner table, (b) both sides are large, (c) the join is on equality. If the hash table exceeds work_mem, it spills to disk (batch hash join) — monitor for this in EXPLAIN.
What is the N+1 query problem and how do you solve it in SQL?
N+1 occurs when code issues 1 query to fetch N parent rows, then N separate queries for each parent's children — N+1 total queries. The SQL solution is a single JOIN or a single WHERE parent_id IN (...) query fetching all children at once, then grouping in application code. In analytics SQL, use ARRAY_AGG or STRING_AGG to collapse children into arrays per parent in one query.
How would you calculate a median in SQL without a MEDIAN() function?
Use PERCENTILE_CONT: SELECT PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY salary) FROM employees. If unavailable, use: SELECT AVG(salary) FROM (SELECT salary, ROW_NUMBER() OVER (ORDER BY salary) rn, COUNT(*) OVER () cnt FROM employees) t WHERE rn IN (FLOOR((cnt+1)/2), CEIL((cnt+1)/2)). The ordered-set aggregate approach is cleaner and available in PostgreSQL, Oracle, and SQL Server.
